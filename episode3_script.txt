《Hermes Agent 架构拆解》第三集：Skills、Memory、Cron 与多平台 Gateway

来到第三集，也是这个系列的收官。前两集我们讲了 Hermes 的大脑和身体：主循环、工具系统、执行后端。今天我们讲“成长机制”，也就是它为什么不是一次性脚本，而是一个可以长期协作的 agent 系统。

这一集有四个关键词：Skills、Memory、Cron、Gateway。

先说 Skills。很多 agent 把“知识”直接堆在系统提示词里，越堆越大，最后上下文膨胀。Hermes 的做法是把程序性知识外置成 Skill 文档，并按需加载。你可以把 Skill 理解成“可检索、可演化的操作手册”。

官方文档强调了 progressive disclosure，也就是分层读取。
第 0 层是技能目录，先看到有哪些能力；
第 1 层再按名字读取某个 Skill 的正文；
第 2 层只在需要时读取该 Skill 的 references、templates、scripts 等附属文件。

这个设计有两个好处：第一，省上下文；第二，技能可以被独立版本化和维护。Skill 通常以 SKILL.md 为入口，带 YAML frontmatter，再加正文步骤说明。Hermes 还支持从 Hub 安装技能，并做安全扫描和信任分级，这比“复制粘贴提示词”工程化得多。

再说 Memory。Hermes 明确把长期记忆分成两类：
MEMORY.md 偏系统事实和稳定约束；
USER.md 偏用户画像和偏好。

而且容量是硬约束，不是无限笔记本。文档给的默认上限是：MEMORY 大约 2200 字符，USER 大约 1375 字符。为什么要这么抠？因为这些内容会进入系统前缀，你让它无限增长，模型每轮都在为历史垃圾付费。

Memory 工具支持 add、replace、remove 三种操作，并做安全扫描，防止把注入指令或敏感数据写成“永久事实”。这里体现了一个很成熟的理念：记忆不是日志，记忆是高价值、低体积、可复用的信息。

那过去的细节去哪了？Hermes 用 session_search 去查 SQLite FTS5 的历史会话索引。也就是“把细节放数据库，需要时检索；把结论放记忆，随会话注入”。这是成本和效果都比较平衡的方案。

第三个关键词是 Cron。Hermes 的 cron 不是传统 shell cron。传统 cron 是“到点跑命令”，Hermes cron 是“到点启动一个新 agent 会话，执行一段任务提示词，并可投递到平台”。

任务定义支持 create、update、pause、resume、run、remove 这类生命周期动作；调度格式支持相对时间、固定间隔、cron 表达式和 ISO 时间；执行时是独立上下文，不继承你当前聊天窗口。你还可以给 job 绑定 Skills，让定时任务也能复用程序性知识。

这是个非常关键的分界线：从“被动问答助手”到“可计划执行的自动化代理”。

第四个关键词是 Gateway。Hermes 的 gateway 是一个长驻进程，负责接不同平台适配器，比如 Discord、Telegram、Slack、WhatsApp、Signal、甚至更多。核心不是“接了多少平台”，而是“多平台共用同一套 agent 内核和状态策略”。

官方文档里你会看到几个工程细节：
一是安全默认拒绝，需要 allowlist 或配对机制；
二是会话管理有重置策略，比如每日重置和空闲超时；
三是用户可以中断正在进行的任务，系统会传播取消信号到工具执行层；
四是支持后台任务与状态反馈，避免聊天界面被长任务阻塞。

到这里你会发现，Hermes 不是把 LLM API 封装一下那么简单，它已经具备了“操作系统味道”：
有内核循环，有工具总线，有权限闸门，有持久化状态，有计划任务，有多终端接入。

如果你想自己做一个生产级 agent，我建议直接借鉴这四个实现哲学：
第一，技能外置且分层加载，别把所有知识塞进常驻 prompt；
第二，记忆要稀缺化管理，短而准；
第三，自动化任务要 agent-native，不要只会跑 shell；
第四，平台接入和核心推理解耦，保证一致行为。

最后做个三集总复盘。
第一集我们讲了 Hermes 的统一主循环，理解了它如何把 Prompt、Provider、Tool 串成闭环。
第二集我们讲了工具体系、执行后端和 MCP，看到它如何扩展能力同时控制风险。
第三集我们讲了 Skills、Memory、Cron、Gateway，看到它如何跨会话学习、跨平台工作、按计划执行。

一句话收尾：Hermes 的价值，不在于“它能回答你什么”，而在于“它能不能稳定地替你完成工作，并且越用越懂你”。

如果你愿意，我下一轮可以继续做一个进阶番外：专门拆 Hermes 的安全模型和高风险命令审批策略，直接对照生产环境里的事故类型来讲，帮助你把自己的 agent 系统做得更抗风险。
